üîÑ Complete Interrupt Processing Flow

1
Hardware Interrupt Occurs
Peripheral device asserts interrupt signal to GIC
2
GIC Processing
GIC Distributor identifies highest priority interrupt and forwards to appropriate CPU interface
3
CPU Exception
CPU receives IRQ/FIQ exception and jumps to vector table
4
Low-level Handler
Assembly code saves context and calls gic_handle_irq()
5
IRQ Acknowledgment
Read GIC IAR register to get interrupt ID and acknowledge
6
Generic IRQ Layer
Convert hardware IRQ to Linux IRQ using IRQ domain
7
Flow Handler
Call appropriate flow handler (handle_level_irq, handle_edge_irq)
8
Device Handler
Execute registered device driver interrupt handler
9
EOI (End of Interrupt)
Write to GIC EOIR register to signal completion
10
Context Restoration
Restore CPU context and return from interrupt

üíª Complete Driver Example

Here's a comprehensive example showing how a device driver interfaces with the GIC through the Linux interrupt subsystem:

#include <linux/module.h> #include <linux/platform_device.h> #include <linux/interrupt.h> #include <linux/of.h> #include <linux/io.h> struct my_device { void __iomem *base; int irq; struct device *dev; }; static irqreturn_t my_device_irq_handler(int irq, void *data) { struct my_device *mydev = (struct my_device *)data; u32 status; // Read interrupt status from device registers status = readl(mydev->base + STATUS_REG); if (!(status & IRQ_PENDING)) return IRQ_NONE; // Not our interrupt // Handle the interrupt dev_info(mydev->dev, "Interrupt handled, status: 0x%x\n", status); // Clear interrupt status writel(status, mydev->base + STATUS_REG); return IRQ_HANDLED; } static int my_device_probe(struct platform_device *pdev) { struct my_device *mydev; struct resource *res; int ret; mydev = devm_kzalloc(&pdev->dev, sizeof(*mydev), GFP_KERNEL); if (!mydev) return -ENOMEM; mydev->dev = &pdev->dev; // Get memory resource res = platform_get_resource(pdev, IORESOURCE_MEM, 0); mydev->base = devm_ioremap_resource(&pdev->dev, res); if (IS_ERR(mydev->base)) return PTR_ERR(mydev->base); // Get IRQ number from platform device (DT or ACPI) mydev->irq = platform_get_irq(pdev, 0); if (mydev->irq < 0) { dev_err(&pdev->dev, "Failed to get IRQ\n"); return mydev->irq; } // Request interrupt line ret = devm_request_irq(&pdev->dev, mydev->irq, my_device_irq_handler, IRQF_TRIGGER_RISING, "my-device", mydev); if (ret) { dev_err(&pdev->dev, "Failed to request IRQ %d: %d\n", mydev->irq, ret); return ret; } platform_set_drvdata(pdev, mydev); dev_info(&pdev->dev, "Driver probed successfully, IRQ: %d\n", mydev->irq); return 0; } static const struct of_device_id my_device_of_match[] = { { .compatible = "vendor,my-device" }, {}, }; MODULE_DEVICE_TABLE(of, my_device_of_match); static struct platform_driver my_device_driver = { .probe = my_device_probe, .driver = { .name = "my-device", .of_match_table = my_device_of_match, }, }; module_platform_driver(my_device_driver); MODULE_LICENSE("GPL");

üéØ Key Concepts Summary

IRQ Domain: Translates hardware interrupt numbers to Linux virtual IRQ numbers

IRQ Chip: Provides hardware-specific interrupt controller operations

Flow Handlers: Generic interrupt processing based on trigger type (level/edge)

Threaded IRQs: Handle heavy processing in kernel threads instead of hard IRQ context

CPU Affinity: Control which CPU cores handle specific interrupts for performance optimization
